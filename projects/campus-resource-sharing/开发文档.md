# campus-resource-sharing 开发文档

## 1. 项目概述

### 1.1 项目背景
随着高校学生消费习惯和资源流转需求的增加，宿舍、社团、毕业季等场景中会产生大量闲置物品，如书籍、电子产品、生活用品、体育器材等。传统线下交易存在信息分散、沟通成本高、可信度低的问题，因此需要建设一个适用于校园内部的闲置物资共享与转让平台。

### 1.2 项目目标
构建一个基于 Spring Boot 和 Vue3 的校园闲置物资共享平台，满足学生用户在校园内进行闲置物品发布、浏览、咨询、收藏、下单、交易管理等需求，同时支持管理员对平台内容进行审核、管理和维护。

### 1.3 适用场景
- 校园内部二手物品转让
- 毕业季闲置清仓
- 社团活动后的物资转让
- 宿舍搬迁时的物品处理
- 同校学生之间的低成本资源流通

### 1.4 目标用户
- 普通用户（学生）：发布闲置、浏览商品、收藏、咨询、下单、管理个人交易
- 管理员：维护平台秩序、管理用户、审核商品、处理举报、发布公告

### 1.5 项目核心价值
- 提高校园闲置资源利用率
- 降低信息不对称与交易成本
- 提供校内可控、轻量、安全的交易环境
- 为课程设计/毕业设计提供具备完整业务闭环的项目案例

### 1.6 校园场景特色
与普通二手平台相比，本项目强调：
- 面向校内用户，交易半径小
- 更适合线下当面交易
- 不强依赖在线支付
- 商品类型更贴近学生日常
- 平台可加入校内公告、审核、举报等校园化管理能力

## 4. 详细功能模块设计

## 4.1 用户认证与账号模块

### 功能目标
实现普通用户和管理员的身份认证、权限鉴别和基础账号维护。

### 核心子功能
- 用户注册
- 用户登录
- 管理员登录
- JWT 鉴权
- 退出登录
- 获取当前用户信息
- 更新个人资料
- 修改密码
- 头像上传
- 用户状态启用/禁用（管理员操作）

### 主要字段/数据项
- 用户名
- 密码
- 昵称
- 头像
- 手机号
- 邮箱
- 性别
- 学院/专业（可选）
- 状态
- 角色
- 注册时间

### 用户操作流程
1. 新用户填写注册信息
2. 系统校验用户名是否重复
3. 密码加密存储
4. 登录成功后签发 JWT
5. 前端保存 token
6. 后续请求通过 token 访问用户信息和业务接口

### 前端页面组成
- 注册页
- 登录页
- 个人资料页
- 修改密码页

### 后端接口需求
- `/api/auth/register`
- `/api/auth/login`
- `/api/auth/logout`
- `/api/user/profile`
- `/api/user/update`
- `/api/user/password`
- `/api/user/avatar`

### 业务规则
- 用户名唯一
- 密码需加密存储
- 禁用用户不可登录
- 管理员与普通用户登录入口可以区分，也可后端统一返回角色信息

### 异常处理
- 用户名已存在
- 密码错误
- token 失效
- 用户不存在
- 上传头像格式不合法

### 权限控制
- 普通用户只能修改自己的资料
- 管理员可查看和管理所有用户状态

## 4.5 收藏与意向模块

### 功能目标
方便用户保存感兴趣商品，提升交易转化率。

### 核心子功能
- 收藏商品
- 取消收藏
- 查看我的收藏

### 是否推荐加入购物车
不建议在初版中实现传统电商购物车。  
原因：
- 校园二手交易通常是一物一卖，库存通常为 1
- 更适合“收藏 + 咨询 + 下单/预约”的轻量模式
- 能有效降低系统复杂度

### 主要字段/数据项
- 用户ID
- 商品ID
- 收藏时间

### 用户操作流程
点击收藏 → 系统保存收藏记录 → 在“我的收藏”中查看 → 可进入详情继续咨询或下单

### 前端页面组成
- 收藏按钮
- 我的收藏页

### 后端接口需求
- `/api/favorite/add`
- `/api/favorite/remove/{goodsId}`
- `/api/favorite/list`

### 业务规则
- 同一用户不能重复收藏同一商品

### 异常处理
- 商品不存在
- 未登录不可收藏

### 权限控制
- 仅登录用户可收藏

## 4.6 订单/交易模块

### 功能目标
实现适合校园场景的轻量交易闭环。

### 推荐交易流程
浏览商品 → 咨询确认 → 提交订单/预约 → 卖家确认 → 线下当面交易 → 买家确认完成

### 是否建议接入在线支付
不建议在初版接入在线支付，原因如下：
- 课程设计/毕业设计实现复杂度较高
- 涉及支付安全、退款、第三方接口审批
- 校园二手交易通常更适合线下验货与面交
- 可通过订单状态管理完成交易闭环

### 订单状态设计
- 待确认
- 已确认
- 已取消
- 交易中
- 已完成
- 已关闭

### 买卖双方关键操作
买家：
- 创建订单
- 取消订单
- 确认完成
- 评价卖家

卖家：
- 确认订单
- 拒绝订单
- 标记交易中
- 确认售出

管理员：
- 查看异常订单
- 手动关闭订单
- 处理投诉订单

### 主要字段/数据项
- 订单编号
- 商品ID
- 买家ID
- 卖家ID
- 成交价格
- 状态
- 备注
- 创建时间
- 完成时间

### 前端页面组成
- 订单创建弹窗
- 我的购买订单页
- 我的出售订单页
- 订单详情页

### 后端接口需求
- `/api/order/create`
- `/api/order/buyer/list`
- `/api/order/seller/list`
- `/api/order/detail/{id}`
- `/api/order/cancel/{id}`
- `/api/order/confirm/{id}`
- `/api/order/complete/{id}`

### 业务规则
- 一个商品成交后不可重复售卖
- 买家不能购买自己的商品
- 已下架商品不可创建订单
- 已完成订单不可修改

### 异常处理
- 商品已售出
- 订单状态不允许当前操作
- 越权操作订单

### 权限控制
- 买家只能查看自己的购买订单
- 卖家只能查看与自己出售有关的订单
- 管理员可查看全部订单

## 6. 业务流程设计

## 6.1 用户注册登录流程
1. 用户进入注册页
2. 输入用户名、密码等信息
3. 后端校验并创建账号
4. 用户登录
5. 后端验证密码，签发 JWT
6. 前端保存 token 并跳转首页
7. 后续接口请求携带 token

## 6.2 发布物品流程
1. 用户登录
2. 进入发布页面
3. 填写标题、价格、分类、描述、联系方式
4. 上传商品图片
5. 提交表单
6. 后端保存商品和图片信息
7. 返回发布成功结果
8. 商品进入上架状态或待审核状态

## 6.3 浏览并购买物品流程
1. 用户浏览商品列表
2. 使用搜索和筛选定位商品
3. 点击进入详情页
4. 查看商品介绍、留言、卖家信息
5. 发起咨询或直接提交订单

## 6.4 收藏与咨询流程
1. 用户在商品详情页点击收藏
2. 系统记录收藏关系
3. 用户可发送留言咨询
4. 卖家在消息模块中查看并回复

## 6.5 订单生成与完成流程
1. 买家点击下单/预约
2. 系统创建待确认订单
3. 卖家确认订单
4. 双方线下约定时间地点
5. 买家确认交易完成
6. 系统更新订单状态为已完成
7. 商品状态改为已售出

## 6.6 举报与审核流程
1. 用户发现违规商品或用户
2. 填写举报原因并提交
3. 后端保存举报记录
4. 管理员在后台查看举报
5. 管理员审核并给出处理结果
6. 系统更新举报状态并执行处罚操作

## 6.7 管理员审核商品流程（推荐）
1. 用户发布商品
2. 商品进入待审核状态
3. 管理员在后台查看商品信息
4. 审核通过后商品上架
5. 审核拒绝则记录原因并通知发布者

## 6.8 管理员处理违规内容流程
1. 管理员收到举报或巡检发现问题
2. 查看商品、用户、举报内容
3. 决定下架商品、删除内容、警告或禁用用户
4. 保存处理记录

## 7.3 表关系说明
- 用户与商品：一对多，一个用户可发布多个商品
- 商品与分类：多对一，多个商品归属一个分类
- 商品与图片：一对多，一个商品可有多张图片
- 用户与收藏：多对多，通过收藏表关联
- 商品与订单：一对多，一个商品理论上只允许一个有效成交订单，历史上可保留尝试订单
- 用户与订单：一个用户可作为买家或卖家参与多个订单
- 用户与留言：一对多
- 用户与举报：一对多
- 订单与评价：一对一或一对多（若双向评价可扩展）

## 9. 权限与安全设计

## 9.1 JWT 登录认证流程
1. 用户登录成功后，后端生成 JWT
2. 前端将 token 存储在 localStorage 或 pinia 持久化中
3. 每次请求通过请求头 `Authorization: Bearer token` 携带
4. 后端拦截器统一校验 token 合法性
5. token 失效后返回未登录状态，前端跳转登录页

## 9.2 前端路由守卫设计
- 用户端需要登录的页面：发布商品、个人中心、我的订单、我的收藏等
- 管理端页面统一增加管理员身份校验
- 若未登录或角色不匹配，则跳转对应登录页

## 9.3 接口鉴权策略
- 公共接口无需 token
- 用户接口必须校验用户身份
- 管理员接口必须校验角色为 admin
- 对涉及资源归属的接口必须校验所属关系，避免越权

## 9.4 密码加密建议
- 使用 BCryptPasswordEncoder
- 不允许明文存储密码
- 修改密码时需校验旧密码

## 9.5 防越权设计
- 编辑商品时校验商品所属 user_id
- 查看订单时校验 buyer_id 或 seller_id
- 举报、评价、留言等操作都需校验用户身份

## 9.6 文件上传安全
- 限制图片格式：jpg/png/jpeg/webp
- 限制文件大小
- 生成唯一文件名
- 不允许上传可执行文件
- 保存路径与访问路径分离

## 9.7 基本数据校验
- 使用 `@NotBlank`、`@NotNull`、`@Min`、`@Length`
- 价格非负
- 内容长度限制
- 枚举状态值严格校验

## 9.8 防重复提交与非法参数
- 创建订单时校验商品状态
- 收藏表增加唯一索引避免重复收藏
- 下单接口可增加幂等校验
- 使用统一异常捕获，避免敏感错误信息泄露

## 9.9 管理员操作日志
建议预留管理员操作日志表，用于记录：
- 登录行为
- 下架商品
- 禁用用户
- 审核举报
- 删除公告
